# Skill: C 语言编程规范

## 技能说明

本技能定义了一套统一的 C 语言编程规范，涵盖**排版规则**、**注释规范**、**命名规则**、**头文件规范**及**宏定义规范**。当用户要求**检查代码规范性**、**格式化 C 代码**或**编写符合规范的 C 代码**时，AI 必须严格遵循以下规则。

## 何时触发

当用户提出以下类型请求时自动激活：

**代码检查场景：**
- "帮我检查这段 C 代码是否符合规范"
- "这段代码有什么不规范的地方"
- "代码审查"

**代码格式化场景：**
- "帮我格式化这段 C 代码"
- "按照规范整理这段代码"

**编写新代码场景：**
- "帮我写一个 C 函数"
- "用 C 实现 xxx 功能"
- 任何涉及 C 代码编写的场景
- 用 C 语言帮我写一个 xxx 程序

## 强制格式规范

### 1. 缩进规范

- 缩进使用 **4 个空格键**。
- `case` 语句下的处理语句也要遵从缩进要求。
- 程序块的分界符（大括号 `{` 和 `}`）中，左花括号 `{` 跟着语句块，右花括号 `}` 独占一行。
- `if`、`for`、`do`、`while` 的执行语句部分无论多少都要加括号 `{}`，且与左花括号在同 1 行，与花括号之间间隔一个空格。

【**正确示例**】

```c
for (...) {
    ... // program code
}

if (...) {
    ... // program code
}

while (...) {
    ...
}

do {
    ...
} while (...);

void example_fun(void) {
    ... // program code
}

switch (variable) {
    case value1:
        ...
        break;
    default:
        ...
        break;
}
```

### 2. 空行规范

- 相对独立的程序块之间、变量说明之后必须加空行。
- 在每个函数定义结束之后都要加 1 个空行。
- 头文件的最后一行为空行，以消除 GCC 警告。

### 3. 行长与换行规范

- 不允许把多个短语句写在一行中，一行只写一条语句。
- 一行程序以小于 80 字符为宜，不要写得过长。
- 对于较长的语句（> 120 字符）要分成多行书写。长表达式在低优先级操作符处划分新行，操作符要放在新行之首，新行进行适当缩进。

### 4. 空格规范

- 逗号、分号只在后面加空格。
- 双目操作符前后加空格。
- 单目操作符（`!`、`~`、`++`、`--`、`&` 地址运算符）前后不加空格。
- `->`、`.`、`[]` 前后不加空格。
- `if`、`for`、`while`、`switch` 等关键字与后面括号间加空格；函数名之后不留空格，紧跟左括号 `(`。
- 左括号 `(` 向后紧跟，右括号 `)`、逗号 `,`、分号 `;` 向前紧跟，紧跟处不留空格。

### 5. 其他排版规范

- 宏定义、函数声明集中放在头文件或文件头；全局变量定义和声明只能放在文件头。
- 文件内顺序：包含的头文件、局部变量定义、全局变量定义、全局变量声明、内部函数声明、外部函数声明。
- 引用多个头文件时，尖括号（`<>`）引用的系统/库头文件在前，双引号（`""`）引用的项目头文件在后。
- 函数内部变量定义集中放在函数开始处，禁止在函数执行语句中随处定义或声明。

## 编码安全规范

### 1. 变量初始化规范

- 局部变量在定义时必须赋初始值，禁止定义未初始化的变量。

【**正确示例**】

```c
int fd = -1;
char *buf = NULL;
int ret = 0;
```

【**错误示例**】

```c
int fd;                        /* 禁止：未初始化 */
char *buf;                     /* 禁止：未初始化 */
```

### 2. 返回值检查规范

- 调用有返回值的函数时，必须使用变量接收返回值，禁止直接丢弃函数返回值。
- 接收返回值后必须对其进行检查，后续逻辑仅在检查通过后执行。

【**正确示例**】

```c
int ret = read_file(fd, buf, size);
if (ret < 0) {
    /* 错误处理 */
    return -1;
}
/* 正常逻辑 */
```

【**错误示例**】

```c
read_file(fd, buf, size);      /* 禁止：未接收返回值 */
```

### 3. 内存管理规范

- 使用 `malloc`、`calloc`、`realloc` 申请内存后，必须检查返回值是否为 `NULL`，禁止不经检查直接使用。
- 动态分配的内存在不再使用时，必须调用 `free` 释放，防止内存泄漏。

【**正确示例**】

```c
char *buf = (char *)malloc(1024);
if (buf == NULL) {
    /* 内存分配失败处理 */
    return -1;
}
/* 使用 buf */
free(buf);
buf = NULL;
```

【**错误示例**】

```c
char *buf = (char *)malloc(1024);
/* 禁止：未检查 buf 是否为 NULL 就直接使用 */
memset(buf, 0, 1024);
/* 禁止：使用完毕后未调用 free 释放内存 */
```

### 4. 函数编写规范

#### 4.1 参数检查

- 若参数中包含指针，必须在函数入口处检查指针是否为 `NULL`，禁止不经检查直接解引用。

【**正确示例**】

```c
int process_data(const char *buf, int len) {
    if (buf == NULL) {
        return -1;
    }
    /* 正常处理逻辑 */
    return 0;
}
```

【**错误示例**】

```c
int process_data(const char *buf, int len) {
    /* 禁止：未检查 buf 是否为 NULL 就直接使用 */
    memcpy(dest, buf, len);
    return 0;
}
```

#### 4.2 主函数版本信息

- `main` 函数中**必须**打印程序版本信息，便于问题追踪和版本确认。
- 版本信息宏（如 `GIT_VERSION`、`GIT_PATH`）由 Makefile 通过 `-D` 选项注入，C 代码中必须使用 `#ifndef` 提供 `"unknown"` 兜底定义，确保无 Makefile 时仍可编译。
- 版本信息应在程序启动后、业务逻辑之前打印，格式为单行输出：`** [函数名:行号][Build Info] 日期 时间, git: 版本, path: 路径 **`，使用 `__func__`、`__LINE__`、`__DATE__`、`__TIME__` 预定义宏。

【**正确示例**】

```c
#ifndef GIT_VERSION
#define GIT_VERSION "unknown"
#endif

#ifndef GIT_PATH
#define GIT_PATH "unknown"
#endif

int main(void) {
    /* 打印版本信息 */
    printf("** [%s:%d][Build Info] %s %s, git: %s, path: %s **\n",
           __func__, __LINE__, __DATE__, __TIME__, GIT_VERSION, GIT_PATH);

    /* 业务逻辑 */
    ...
}
```

【**错误示例**】

```c
/* 禁止：直接使用宏而不提供兜底定义，无 Makefile 时编译报错 */
int main(void) {
    printf("Version: %s\n", GIT_VERSION);
    ...
}
```

## 注释规范

### 1. 注释风格

- 统一使用 **doxygen** 注释风格。
- 注释语言尽可能使用中文。
- 注释应放在被注释代码的上方或右方，如放于上方则必须与上面的代码用空行隔开。
- 注释与所描述内容进行同样的缩进。
- 修改代码同时更新注释，不再有用的注释要删除。
- 多行注释使用 `/* ... */` 格式，不得使用 `//` 进行连续多行注释。

### 2. 文件头注释

文件头部应进行注释，注释必须列出：版权说明、版本号、生成日期、作者、内容、功能、修改日志等。

【**推荐格式**】

```c
/** =====================================================
 * Copyright © sumu. 2022-present. Tech. Co., Ltd. All rights reserved.
 * File name  : 文件名
 * Author     : xxx
 * Date       : YYYY/MM/DD
 * Version    : 
 * Description: 
 * ======================================================
 */
```

### 3. 函数头注释

函数头部应进行注释，注释要符合 doxygen 风格，至少要包含 brief、param、return 字段，对于有额外需要注意的或者要说明的可以使用 note 字段。

【**推荐格式**】

```c
/** @fn const char *TestClass::func1(char c, int n)
 *  @brief <这里填写函数的摘要>
 *  @param[in]  c 输入参数描述.
 *  @param[out] n 输出参数描述.
 *  @return 返回描述，例如，-1表示失败，0表示成功
 */
```

### 4. 成员/变量/宏注释

- 成员注释可使用 `/* */`、`//` 或 `///<` 格式。
- 宏定义注释可使用 `/* */` 或 `//` 格式。
- 全局变量要有详细注释，包括功能、取值范围、存取函数及注意事项。

【**示例**】

```c
long lValue; /* 成员描述 */
#define MIN(x,y) ((x)>(y)?(y):(x)) /* 注释 */
```

### 5. 自注释与层次注释

- 通过对函数、变量、结构等正确命名及合理组织代码，使代码成为自注释的。
- 在代码的功能、意图层次上进行注释，提供有用、额外的信息。避免无意义的重复注释。
- 在程序块结束行右方加注释标记（如 `/* end of if */`），便于阅读。

## 命名规范

### 1. 基本原则

- 标识符命名要清晰、明了，有明确含义。
- 标识符长度不得超过 32 个字符。
- 禁止取单个字符作为变量名（`i`、`j`、`k` 作为局部循环变量除外）。
- 命名风格必须与系统风格一致：**函数采用全小写加下划线**，**变量采用大小写混排**。不要使用大小写与下划线混排（如 `Add_User` 不允许，`add_user`、`AddUser`、`g_AddUser` 允许）。
- 宏定义、枚举定义全部使用大写，多个单词用下划线分割。
- `typedef` 定义的类型全部使用大写。
- 结构体名由大写字母、数字和下划线组成，不允许小写字母，多个单词用下划线分割。
- 变量名使用名词或形容词+名词。

### 2. 反义词组

- 用正确的反义词组命名互斥变量或相反动作的函数（如 `add/remove`、`begin/end`、`get/release` 等）。

### 3. 其他

- 除非必要，不要用数字或奇怪字符定义标识符。

## 头文件规范

- 头文件必须包括文件信息声明，并使用 ifndef/define/endif 预处理，防止头文件重复引用，宏一般就是 `__文件名_H`。

【**示例**】

```c
#ifndef __XXX_H
#define __XXX_H

#endif
```

- 头文件要兼容 C++，要添加 `extern "C"` 结构，**确保在 C++ 代码中能使用 C 代码**。

```c
#ifdef __cplusplus
extern "C" {
#endif
/*具体代码*/
#ifdef __cplusplus
}
#endif
```

## 宏定义规范

- 用宏定义表达式时，要使用完备的括号。
- 前后相邻的宏定义的表达式上下对齐。

【**示例**】

```c
#define RECTANGLE_AREA(a, b) ((a) * (b))
#define MAX_CNT              (12)
```

## 行为规范

### 代码检查

当用户要求检查代码规范性时：

1. 逐一对照本规范进行检查。
2. 指出不符合规范的具体位置和原因。
3. 提供修改建议。

### 代码编写

当用户要求编写 C 代码时：

1. 优先应用本规范的所有格式要求。
2. 为文件添加标准文件头注释。
3. 为公共函数添加 doxygen 风格函数头注释。
4. 确保命名符合规范要求。
5. `main` 函数中必须包含版本信息打印逻辑，使用 `#ifndef` 为 `GIT_VERSION`、`GIT_PATH` 等宏提供兜底定义。
